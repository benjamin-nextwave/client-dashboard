## Phase Details

### Phase 9: News Authoring & Schema
**Goal**: Operators can author, manage, and publish multilingual news items, with all schema and storage groundwork in place to serve them to clients
**Depends on**: Phase 1 (multi-tenancy foundation), Phase 2 (operator admin pattern)
**Requirements**: NEWS-01, NEWS-02, NEWS-03, NEWS-04, NEWS-05, NEWS-06
**Justification**: All operator-facing authoring requirements share the same admin UI surface, the same `news_items` table, and the same Supabase Storage bucket for images. Bundling them produces tight cohesion: every requirement directly contributes to "operator can broadcast a multilingual announcement." Splitting would scatter database migrations across phases. The `news_dismissals` table is also created here so Phase 10 has nothing to wait on schema-wise.
**Success Criteria** (TRUE at close):
  1. ✓ An operator can open the operator admin panel, create a news item with an image, and fill in title + body in Dutch, English, and Hindi (devanagari) variants
  2. ✓ An operator can edit any field of any language variant of an existing news item and the change persists
  3. ✓ An operator can preview a news item in any of its three language variants before publishing
  4. ✓ An operator can publish a draft and withdraw a published item from the admin list view
  5. ✓ The admin list view shows every news item with its status (draft / published / withdrawn) and creation/publish timestamps

## Cross-cutting constraints (truths that appeared in 2+ plans)

**Phase 9 / authoring side:**
- Server-authoritative publish gate: all 6 language fields must be non-empty before status transitions to `published` (newsPublishSchema refine + DB re-read in publishNewsItem)
- Soft-delete withdraw: status transitions to `withdrawn` + `withdrawn_at` timestamp; row remains in admin list (NEWS-04, NEWS-05)
- Multilingual content lives on the row (title_nl/en/hi, body_nl/en/hi); operator UI chrome strings live in `operator.news.*` i18n namespace (D-23, D-24)
- Image storage: `news-images` bucket, 2MB, png/jpeg/webp only; `image_path` is server-managed (never read from formData)
- Component reuse: news-preview-modal exports a presentational `NewsContentRenderer` that Phase 10 reuses for the client overlay (D-05)

**Phase 10 / delivery side:**
- Overlay scope is `/dashboard` ONLY (not /inbox or other subroutes) — overlay mounted in dashboard/page.tsx, not in (client)/layout.tsx
- Dismiss path strictly single-button — no Escape, no backdrop click, no X icon (deliberate divergence from standard modal behavior, documented inline in code)
- Sidebar uses standard close conventions (X, Escape, click-outside) — divergence from overlay is intentional
- `dismissNewsItem` uses request-scoped client (NOT admin); reads user_id from `auth.getUser()` server-side; never accepts user_id from client
- Two-query pattern (archive + dismissals) over subquery interpolation; TS Set filter is mechanically equivalent and avoids template-string SQL

## Milestone Summary

### Decisions logged

**Architecture:**
- Two-phase split (operator authoring vs client delivery) chosen over a single phase for tighter cohesion and independently verifiable capabilities; `news_dismissals` table created in Phase 9 alongside `news_items` to remove cross-phase schema dependencies (zero blockers between phases).
- Global broadcast (no `client_id` on news_items) — published news visible to every authenticated client across all tenants. Per-user dismissal granularity via `news_dismissals (user_id, news_item_id)` composite PK.
- Soft-delete withdraw (status enum: draft/published/withdrawn) preserves audit trail; clients see only `status='published'` rows via RLS.
- Image upload optional (nullable `image_path`); operators can broadcast pure-text announcements.

**UX:**
- Single-button dismiss path on the overlay ("Ik heb het gelezen" — no Escape, no backdrop, no X) ensures the user actively acknowledges the message; deliberate divergence from standard modal conventions.
- Overlay queue model (oldest unread first, dismiss-to-advance, gone after last) chosen over single-most-recent or all-at-once-stack — every published item gets a viewing.
- Megaphone is a secondary action visually (outlined icon, neutral colors) — brand-colored CTA reserved for primary actions like RefreshButton.
- Removed unread-count badge during smoke verification (overlay-on-open makes "unread count" always 0 in practice — the badge would always be misleading).

**Implementation:**
- `'use server'` files MUST export only async functions — type re-exports break at runtime in Next.js 15 + Turbopack (Locale bug surfaced and fixed in `040a936`).
- Tailwind 4 keyframe pattern (define in `@theme inline` + `@keyframes`) for animations — no `tailwindcss-animate` dependency added.
- React 19 + Next.js 15 + RHF + soft-nav redirect: form `defaultValues` from `useForm()` does not populate inputs reliably after `redirect()` from a server action; defense-in-depth fix is explicit JSX `defaultValue` on each input + `key={row.id}` on the form (commit `0bdedb2`).
