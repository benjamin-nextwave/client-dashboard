  <action>
Run the Supabase schema push command (use `--linked` consistently — the project deploys against a linked Supabase project, not a local Docker DB):

```bash
npx supabase db push --linked
```

Expected behavior:
- The CLI lists pending migrations: `20260429000002_news_broadcasting.sql`.
- The CLI may prompt for confirmation (Y/N). If the environment is non-interactive, set `SUPABASE_ACCESS_TOKEN` env var or use `--yes` flag where supported.
- The migration applies. Output ends with something like `Finished supabase db push.`

If the push fails with a syntax error in the SQL:
1. Read the error message carefully.
2. Open `supabase/migrations/20260429000002_news_broadcasting.sql`.
3. Fix the issue.
4. DO NOT delete the migration file or rename it — that creates drift.
5. Re-run `npx supabase db push --linked`.

If the push fails with "migration already applied":
- Either the migration ran in a previous attempt — verify by querying the live DB (Task 3 below).
- Or there's a name collision with an existing migration — verify the timestamp `20260429000002` is unique under `supabase/migrations/`.

If the push prompts for an access token:
- Set `SUPABASE_ACCESS_TOKEN` from the local `.env.local` or from the Supabase dashboard.
- Re-run.

If the CLI requires an interactive prompt that cannot be auto-answered AND we have no `--yes` flag, escalate via the autonomous=false flag — operator runs the command manually, confirms success, and pastes the output here.
  </action>

Option B — direct queries (requires DB connection string from `.env.local`):
```bash

```

- Open the project's Supabase Dashboard → Table Editor → confirm `news_items` and `news_dismissals` tables exist.
