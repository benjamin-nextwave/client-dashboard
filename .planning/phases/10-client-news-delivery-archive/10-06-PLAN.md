<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 1: End-to-end manual smoke verification of Phase 10 client news delivery</name>
  <what-built>
Phase 10 is fully wired:
- i18n keys for `client.news.*` exist in nl/en/hi (10-01)
- `dismissNewsItem` server action exists (10-02)
- `news-overlay.tsx` client component exists with single-button dismiss + scroll lock + queue advancement (10-03)
- `news-sidebar.tsx` + `news-megaphone-button.tsx` exist with list↔detail + close affordances + unread badge (10-04)
- `dashboard/page.tsx` fetches archive + dismissals, pre-localizes server-side, resolves image URLs, renders the new components in the header and as a sibling (10-05)

The migration creating `news_items` and `news_dismissals` was deployed in Phase 9 (`20260429000002`). The bucket `news-images` exists. RLS is in place. Phase 9's operator UI is live. Phase 10 needs no schema or storage work.
  </what-built>
  <how-to-verify>
Pre-flight: `npm run dev` is running. Use the local URL printed by Next (typically `http://localhost:3000`). Use TWO test accounts: a real OPERATOR account that you already use for Phase 9 smoke, plus at least ONE test CLIENT account in any tenant. Have the Supabase Studio (DB) tab open in another window so you can inspect `news_items` and `news_dismissals` rows directly.

If you do not yet have a test client, create one in the operator admin or use an existing client whose dashboard you can sign into.

### Step 1 — Setup: clean slate

In Supabase Studio's Table Editor:
1. Open `news_items`. Confirm the table exists. If there are leftover items from Phase 9 smoke, this is fine — but for a clean test, optionally delete or withdraw them first via `/admin/news`.
2. Open `news_dismissals`. Note the current rows. If there are dismissals for the test client's user_id from prior testing, optionally `DELETE FROM news_dismissals WHERE user_id = '<test-client-user-id>'` to start fresh.

### Step 2 — Operator: publish a Phase-10 test item

Sign in as the operator. Navigate to `http://localhost:3000/admin/news`.
1. Click "Nieuw nieuwsbericht" → fill all 6 fields:
   - title_nl: "Phase 10 test (NL)"
   - title_en: "Phase 10 test (EN)"
   - title_hi: "फेज 10 परीक्षण" (or any devanagari content)
   - body_nl: "Dit is een test van de overlay-flow. " repeated to >120 chars to verify preview truncation in the sidebar.
   - body_en: same in English, also >120 chars.
   - body_hi: any devanagari content, also >120 chars.
   - Image: upload any PNG / JPEG / WebP (~200KB). The image will be visible in overlay + detail view.
2. Save the draft, then click "Publiceren". Confirm the status flips to "Gepubliceerd".
3. In Supabase Studio confirm a `news_items` row was inserted with `status='published'`, all 6 lang fields populated, `image_path` set, and `published_at` non-null.

### Step 3 — Client: overlay-on-open (DELIVER-01, DELIVER-02)

Open an INCOGNITO browser window (so the client session is independent from the operator session). Sign in as the test client. Navigate to `/dashboard`.

Expected:
- ☐ A full-screen overlay appears centered, with a semi-transparent dark backdrop.
- ☐ The overlay shows the image (full-width, 16:9), the title, and the body of the test item — in the client's profile language.
- ☐ Body scroll on the underlying dashboard is locked (try scrolling the page behind — should not move).
- ☐ There is exactly ONE button visible inside the overlay: localized "Ik heb het gelezen" / "I have read it" / Hindi equivalent. Brand color matches the client's primary color.
- ☐ There is NO X / close icon anywhere in the overlay.
- ☐ The page-level megaphone button shows a red "1" badge in its top-right corner.

If the title/body language doesn't match expectation: in Supabase Studio's Table Editor, check `profiles.language` for the test client's user_id (`UPDATE profiles SET language = 'en' WHERE id = '<test-client-user-id>';` to switch to English, then reload).

### Step 4 — Client: dismiss-path enforcement (DELIVER-03)

With the overlay still open:
- ☐ Press the **Escape** key. Expected: NOTHING happens. Overlay stays open.
- ☐ Click the dark area OUTSIDE the white card (the backdrop). Expected: NOTHING happens. Overlay stays open.
- ☐ Open browser DevTools → there should be no `keydown` listener registered for Escape that closes this overlay (verify by inspecting `Event Listeners` in the Elements panel on the backdrop div — onClick is absent or no-op).

### Step 5 — Client: dismiss + persistence (DELIVER-04)

- ☐ Click the "Ik heb het gelezen" button.
- ☐ Expected (the test item is the only unread): the overlay closes, body scroll is restored, the dashboard becomes interactive.
- ☐ The megaphone badge updates: either disappears (if no other unread items) or decrements.
- ☐ In Supabase Studio's `news_dismissals` table, a new row exists: `(user_id = test client's user_id, news_item_id = the published item's id, dismissed_at = now)`.
- ☐ Reload the page. Overlay does NOT re-appear.
- ☐ Sign out, sign back in as the same client, navigate to /dashboard. Overlay still does NOT re-appear (persistent dismissal).

### Step 6 — Client: sidebar archive (ARCH-01..04)

- ☐ ARCH-01: The megaphone button is visible in the page header, IMMEDIATELY to the left of the "Ververs data" button. Visually compare positions.
- ☐ Click the megaphone button.
- ☐ ARCH-02: A right-side panel slides in from the right edge. The dashboard content stays underneath but is dimmed by a backdrop.
- ☐ ARCH-02: The list contains the test item (even though the user dismissed it — the sidebar is the FULL ARCHIVE, not the unread list).
- ☐ ARCH-03: The list item shows: title + a body preview (verify it ends with `…` since the body was >120 chars) + a relative time string (e.g., "zojuist" / "just now" / Hindi equivalent). NO image is shown in the list item.
- ☐ Click the list item.
- ☐ ARCH-04: The same sidebar transitions to a detail view showing the image (full-width), the full title, and the FULL body (no truncation). At the top of the detail view is a "Terug naar overzicht" (or its translation) button.
- ☐ Click "Terug naar overzicht". The list view returns.
- ☐ Click the X close icon in the sidebar header. The sidebar closes.
- ☐ Re-open via the megaphone. Press Escape. The sidebar closes (Escape DOES work for the sidebar — opposite of the overlay).
- ☐ Re-open via the megaphone. Click the backdrop (the dark area to the left of the panel). The sidebar closes.

### Step 7 — Multi-item queue verification (D-28)

- ☐ As the operator: publish a SECOND test item. (Repeat Step 2 with different content.)
- ☐ Optional: as a Postgres admin, `DELETE FROM news_dismissals WHERE user_id = '<test-client-user-id>'` to reset the client's dismissals.
- ☐ As the client: visit /dashboard. The OLDEST unread item shows first.
- ☐ Click "Ik heb het gelezen". The overlay should advance — either show the next unread item without backdrop flash, OR close if that was the last unread. Confirm the queue advances cleanly.
- ☐ If two items were unread: a `news_dismissals` row exists for the FIRST dismissed item; the SECOND is now the active overlay item. Click "Ik heb het gelezen" again. Both rows now exist; overlay closes.

### Step 8 — Withdrawal propagation (DELIVER-05)

- ☐ As the operator: open `/admin/news`, click the published test item, click "Intrekken" / "Withdraw" (NOT delete). Confirm status flips to "Ingetrokken".
- ☐ As the client (in another window — DO NOT reload yet): open the sidebar via the megaphone. Note that the still-open sidebar may still show the item until reload (acceptable per SPEC: "within one page reload").
- ☐ Reload `/dashboard`. The withdrawn item should be GONE from the sidebar list AND not present in the overlay queue.

### Step 9 — Scope: overlay does NOT fire on subroutes

- ☐ Republish or create a fresh unread item (so the overlay would fire for this client).
- ☐ As the client: navigate directly to `/inbox` (or `/voorvertoning`, `/dnc`, `/mijn-campagne`, `/feedback`) WITHOUT visiting `/dashboard` first. Confirm: NO overlay appears.
- ☐ Confirm the megaphone button is also NOT in the header on these subroutes (it's specific to `/dashboard` per SPEC).
- ☐ Navigate to `/dashboard`. Overlay appears (as expected — the unread item is still unread).

### Step 10 — Tri-locale rendering

For the same item:
- ☐ In Supabase Studio: `UPDATE profiles SET language = 'nl' WHERE id = '<client-user-id>'`. Reload `/dashboard`. Title/body in Dutch.
- ☐ `UPDATE profiles SET language = 'en' WHERE id = '<client-user-id>'`. Reload. Title/body in English.
- ☐ `UPDATE profiles SET language = 'hi' WHERE id = '<client-user-id>'`. Reload. Title/body in Hindi devanagari.
- ☐ The dismiss button label, megaphone aria-label, sidebar title, sidebar empty state, and relative time strings ALL track the locale change.

### Step 11 — RLS sanity (T-1 mitigation)

In Supabase Studio's SQL editor (as a non-admin), simulate an attacker:
- ☐ As the test client (cookied session), open the browser DevTools → Network tab. Click "Ik heb het gelezen" once. Note the server-action POST. The payload contains the news_item_id but NOT a user_id (the action reads user_id server-side from the auth session — verified by inspecting the request body).
- ☐ Confirm in `news_dismissals` that the inserted row's `user_id` matches the test client's auth.uid (NOT some forged value).

### Step 12 — Cleanup

- ☐ Delete the test items via `/admin/news` (operator). Cascading FK on news_dismissals cleans up the dismissals automatically.
